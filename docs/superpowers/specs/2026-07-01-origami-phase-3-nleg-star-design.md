## 1. 배경 / 문제

Phase 1~2로 "동물 이름 → LLM 트리 → CP → 3D 접기" 전체 파이프라인은 동작하지만,
엔진이 **"잎 3개 + 분기점 1개(트라이포드)"** 에 처음부터 끝까지 하드코딩돼 있다:

- LLM 도구 `emit_animal_tripod`: 다리를 정확히 3개로 강제(`minItems:3, maxItems:3`)
- `buildStarTripod`: 잎 3개 + 분기점 1개만 허용, SSS **닫힌 공식**으로 삼각형 작도
- `buildRabbitEarMolecule`: 삼각형 1개만 처리
- `assembleFold`: 정점 7·변 12·면 6개로 고정

결과적으로 **어떤 동물을 입력해도 "가운데서 3방향으로 뻗은 삼각형"** 으로 수렴한다.
사용자가 "학을 넣어도 사자를 넣어도 똑같은 삼각형"이라는 불만을 제기했고, 이 표현력 한계가 확인되었다.

## 2. 목표 / 비목표

### 목표

- 엔진을 **다리 N개(3 ≤ N ≤ 6)** 로 일반화한다. N=3은 특수 케이스로 계속 동작.
- LLM이 동물을 **4~6개 주요 부위**로 표현하도록 도구/프롬프트를 넓힌다.
- 서로 다른 동물이 **서로 다른 다각형 별 모양 CP**로 나오게 한다.
- 생성된 모든 CP가 (1) 유효 FOLD, (2) Origami Simulator에서 발산 없이 접힘, (3) 결정적(같은 트리 → 같은 FOLD).

### 비목표 (후속 Phase)

- **몸통축 / river(강) 처리** — 머리-몸통-꼬리 스파인. TreeMaker에서 가장 어려운 부분. 다음 Phase.
- 비대칭 스파인, 회전 대칭.
- 오목(non-convex) 패킹 영역, 다중 내부 분기점.
- 4잎 초과의 완전 최적(최대 스케일) 패킹 보장 — "결정적이고 유효하게 접힘"을 우선.

## 3. 접근 개요

파이프라인 4단계를 "N=3 고정" → "N≥3 일반"으로 확장한다.

| 단계 | 현재 (N=3) | Phase 3 (N=3~6) |
|---|---|---|
| ① LLM 도구 | 다리 정확히 3개 | 다리 4~6개 (엔진은 N≥3 일반 처리) |
| ② 잎 배치 | SSS 삼각형 닫힌 공식 | 순환 다각형 작도 (1D 근찾기, 닫힌 형식) |
| ③ 몰리큘 | rabbit-ear(삼각형 1개) | 단일 허브 방사형 몰리큘 (tripod를 N개로 직접 일반화) |
| ④ FOLD 조립 | 정점7·변12·면6 고정 | 정점·변·면 동적 생성 |

### 3.1 트리 위상

- **단일 중심 분기점** `branch` + 다리 N개, 모든 엣지가 `branch → leaf`.
- 잎 i, j 사이의 트리 거리 = `e_i + e_j` (두 잎 원이 접한다고 가정, Lang SoCG96 Theorem 2).
- N=3이면 세 쌍이 동시에 접할 수 있어 닫힌 공식(현행) 성립. N≥4는 과결정 → 최적화 필요.

## 4. 컴포넌트 설계

### 4.1 잎 배치 — 순환 다각형(cyclic polygon) 작도 (신규, ⭐ 핵심)

**문제**: 각 잎 i에 반지름 `e_i` 원을 놓되, 인접(순환 순서상) 잎이 서로 접하고
(`|p_i − p_{i+1}| = e_i + e_{i+1}`), 비인접 쌍은 겹치지 않으며, 전체가 볼록하게 배치되도록.

**선택한 접근 — 순환 다각형 닫힌-형식 작도(반복 이완 NLP 아님)**:

인접 거리 `d_i = e_i + e_{i+1}`(순환, N개)를 변 길이로 갖는 **원에 내접하는(cyclic) 다각형**은
유일하다. 외접원 반지름 `R`에 대한 1차원 단조 방정식으로 구한다:

```
Σ_i 2·arcsin( d_i / (2R) ) = 2π   →  R을 이분법(bisection)으로 근찾기
```

- `R`을 찾으면 각 변의 중심각 `θ_i = 2·arcsin(d_i/2R)`로 정점을 원 둘레에 순서대로 배치.
- 각 잎 i는 그 정점, 반지름 `e_i`. 인접은 작도상 정확히 접함(active).
- **N=3이면 이 순환삼각형 = 기존 SSS 삼각형(외접원)** — Phase 1과 동일 도형(좌표계만 다름).
- 근찾기는 `R`에 대해 좌변이 단조감소라 **강건·결정적**(`Math.random()` 없음).

**기각한 대안**:
- 반복 이완(relaxation)·자체 NLP: 지역최소·수렴 튜닝·결정성 확보 부담. 순환 다각형은 닫힌 형식이라 불필요.
- C 솔버 WASM: 빌드 복잡도 과함.

**결정성 주의**: 근찾기 반복횟수·허용오차 고정, 정점 배치 순서 고정 → 같은 트리는 항상 같은 좌표.

### 4.2 몰리큘 — 단일 허브 방사형(single-hub radial) (⭐ tripod 직접 일반화)

배치된 순환 다각형에서 CP를 채운다. **거싯(gusset) 없는 단일 허브** 구조로, 기존 tripod(N=3)를
그대로 일반화한다:

- **허브 H**(분기점) = 잎 위치의 가중 평균 `H = Σ_i p_i·(S − e_i) / ((N−1)·S)`, `S = Σ e_i`.
  N=3이면 기존 `buildRabbitEarMolecule`의 가중 incenter 공식과 동일.
- **접선점 t_i** = 인접 잎 사이 변 위, 비율 `e_i : e_{i+1}` 지점(인접이 접하므로 잘 정의됨).
- **크리스**: H→leaf_i (전부 **M**, N개) + H→t_i (기본 **V**, 단 첫 번째만 **M**으로 뒤집음, N개).
  → M = (N+1)개, V = (N−1)개, **|M−V| = 2 (Maekawa, 모든 N에서 성립)**.
  N=3이면 4M·2V로 기존 rabbit-ear와 정확히 일치.
- **면**: `[leaf_i, t_i, H]`, `[t_i, leaf_{i+1}, H]` → 총 **2N개 삼각형 면**.
- **경계**: 각 변을 접선점에서 둘로 나눈 **2N개 경계 변(B)**.

**정확도 한계(정직한 명시)**: 이 단일 허브 몰리큘은 **접선 다각형(incircle 존재)일 때만 정확히
평평접기(flat-foldable)** 다. N=3(삼각형)은 항상 성립, 대칭 N-gon도 성립. 비대칭·비접선
다각형에서는 허브에서 Kawasaki가 정확히 안 맞아 **엄밀히는 평평접기가 아니지만**, Origami
Simulator의 물리 이완이 N-flap 베이스로 접어 준다(발산 없이). 이는 취미용 단순화 엔진의
의도된 트레이드오프이며, 정통 universal molecule의 거싯 처리는 후속 Phase로 남긴다.

- **비인접 겹침 / 퇴화 케이스**: 근찾기 실패(`d_i ≥ 2R` 불가) 또는 비인접 원 겹침이 감지되면
  **깨진 FOLD를 내보내지 않고** 예외를 던져 기존 에러 UI로 실패 처리(상위 스펙 §5).

### 4.3 FOLD 조립 (일반화)

- `assembleFold`을 고정 7정점 → **동적 정점/변/면 생성**으로 재작성.
- 경계 변(B), 내부 크리스(M/V), `edges_foldAngle`(M=−180, V=180, B=null, F=0) 채우기는 현행 컨벤션 유지.
- Maekawa(|M−V|=2)·Kawasaki 제약을 몰리큘 단위로 만족.

### 4.4 LLM 도구 / 프롬프트

- `emit_animal_tripod` → **`emit_animal_star`** 개명. `legs`를 `minItems:4, maxItems:6`.
- 프롬프트: "실제 동물의 주요 부위(머리·꼬리·다리·날개) **4~6개**로. 대칭 동물이면 대칭 길이 권장(강제 아님)."
- `tripodInputToTree` → **N개 다리 처리**로 일반화(노드/엣지 동적 생성).
- 검증(백엔드): 잎 개수 4~6, 모든 엣지 길이 > 0, 트리 연결·무사이클(현행 재사용). 실패 시 1회 재시도.

### 4.5 대칭 / 결정성

- **대칭 강제 안 함.** 엔진은 LLM이 준 트리를 그대로 패킹. 대칭은 프롬프트 유도만.
- **결정성**: 고정 초기배치 + 고정 반복 → 같은 트리는 항상 바이트 동일 FOLD. URL 공유의 전제.

## 5. 데이터 모델 영향

- `Tree` / `TreeNode` / `TreeEdge`: 변경 없음(이미 N개 노드 일반). `leafIds` 재사용.
- `StarTripod`(고정 3-튜플) → **`StarPacking`**(신규): `{ branchNodeId, leaves: PackedLeaf[] }`,
  `PackedLeaf = { id, edgeLength, position }`. 3-튜플 타입 제약 제거(길이 N 배열).
- 일반 몰리큘 결과 타입 **`StarMolecule`** 신설: `{ hub, tangentPoints, creases }`. N=3을 포함해
  전 범위를 담당한다(기존 `buildStarTripod`/`buildRabbitEarMolecule`는 제거하고 일반 경로로 대체;
  Phase 1 코드는 git 히스토리와 골든 테스트 값으로 회귀 기준만 남긴다).
- `FoldDocument`: 변경 없음(이미 동적 배열).

## 6. 에러 처리

상위 스펙 §5 표를 계승하고 다음을 명시:

| 실패 지점 | 처리 |
|---|---|
| 순환 다각형 근찾기 실패(`d_i ≥ 2R` 불가) | 예외 → 워커 error 메시지 → 기존 에러 UI + "다시 시도" |
| 비인접 잎 원 겹침(자기교차 베이스) | 예외 → 동일 처리. 깨진 FOLD 절대 미방출 |
| LLM 잎 수 범위 밖(<4 또는 >6) | 검증 실패 → 1회 재시도 → 502 |
| 조작된 URL(엔진이 못 접는 트리) | `decodeTreeFromUrlParam`이 `treeToFold`로 검증(현행) → null → 폼으로 |

## 7. 테스트 전략 (TDD)

Phase 1·2와 동일하게 subagent-driven-development + task별 implementer/reviewer 페어링.

- **패킹 골든값**: 4잎 대칭(정사각형 배치)·5잎 등 알려진 배치와 ε 매치.
- **결정성**: 같은 트리 2회 → 바이트 동일 FOLD.
- **몰리큘**: 볼록 사각형·오각형 골든 CP. 삼각형은 기존 rabbit-ear 회귀 유지.
- **FOLD 유효성**: 모든 N(3~6)에서 스키마 검증 통과, Maekawa/Kawasaki 국소 확인.
- **일반화 회귀**: 기존 학 트라이포드(N=3)가 일반 경로에서도 동일/동등 결과.
- **E2E 수동**: "학"·"사자"·"도마뱀" 입력 → 서로 다른 별 모양 CP → 3D 접힘(운영 URL 시각 확인).

## 10. 리스크 / 미정

- **최대 리스크**: 단일 허브 몰리큘은 비접선 다각형에서 엄밀한 평평접기가 아님 → 시뮬레이터가
  발산 없이 접는지 E2E로 반드시 확인(단위 테스트만으로 불충분). 발산하면 허브 위치/배정 규칙 조정.
- **차선 리스크**: 길이 편차가 큰 트리(한 다리만 매우 긺)에서 비인접 원이 겹칠 수 있음 →
  겹침 검출로 명시적 에러 처리, 골든/스냅샷으로 조기 감지.
- **해소됨**: 잎 배치는 순환 다각형 닫힌-형식 작도로 확정(NLP 불필요). 몰리큘은 tripod의
  단일 허브 구조를 N개로 직접 일반화(거싯 없음)로 확정.
